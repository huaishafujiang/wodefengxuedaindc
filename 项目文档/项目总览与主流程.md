# 项目总览与主流程

这份文档是当前项目的主说明。目标不是把每个历史功能都列一遍，而是把**现在真正该用的主线**讲清楚。

## 1. 项目现在能做什么

项目本质上是一条 STM32G431 频响测量链：

1. 固件输出扫频激励并采集响应。
2. 上位机读取 `omega`、`Magnitude_data`、`Phase_data_rad`。
3. 上位机绘制 Bode 和 Nyquist 曲线。
4. 自动判断滤波器类型、阶次和关键频率。
5. 计算相位裕度、增益裕度和 Nyquist 稳定性。
6. 按控制理论给出校正器建议。
7. 导出报告、CSV 和图片，便于演示或写实验文档。

## 2. 推荐主流程

### 路径 A：接 STM32 实机

1. 打开 `系统分析/app.py` 对应的 Tkinter 上位机。
2. 选择串口。
3. 设置扫频起点、终点、步进和幅值。
4. 点击“开始扫频分析”。
5. 查看图形、右侧摘要和下方详细报告。
6. 必要时切到“自动控制校正”页填写目标指标。
7. 导出 HTML / CSV / 图像。

### 路径 B：导入实测文本

1. 准备文本文件，内容至少包含：

```text
omega=[...]
Magnitude_data=[...]
Phase_data_rad=[...]
```

2. 在“导出设置”页点击“导入文本”。
3. 软件自动完成分析、绘图和报告生成。
4. 若要做控制校正，去“自动控制校正”页填目标参数。

## 3. 当前主模块

| 模块 | 状态 | 作用 |
| --- | --- | --- |
| `系统分析/app.py` | 主入口 | Tkinter 界面、分析调度、报告组织。 |
| `系统分析/filter_analysis.py` | 核心 | 频响特征、类型识别、阶次估计、稳定性分析。 |
| `系统分析/ai_diagnosis.py` | 核心 | AI 辅助解释、候选排序、补扫建议。 |
| `系统分析/control_compensation.py` | 核心 | 自动控制原理校正。 |
| `系统分析/serial_protocol.py` | 核心 | 串口文本协议。 |
| `系统分析/serial_readers.py` | 核心 | 串口线程读取。 |
| `系统分析/smart_resweep_reader.py` | 辅助 | 智能补扫线程。 |
| `系统分析/plotting.py` | 核心 | 曲线绘图。 |
| `系统分析/report_export.py` | 核心 | HTML / CSV 导出。 |

## 4. 现在刻意砍掉或降级的内容

这些东西不是没用，而是**不应该在主流程里跟核心功能抢位置**：

1. 仪器参考校正
   - 以前是测量链补偿。
   - 现在不作为默认入口，避免“校正”一词被混淆。

2. 手动元件偏差诊断
   - 适合做实验型辅助，但不适合主界面默认演示。
   - 保留代码和测试，必要时可重新接回。

3. 原理图批量拟合与维修助手
   - 更像附加分析，不是主仪器的第一层职责。
   - 已从默认主流程中拿掉。

4. PySide6 视觉外壳
   - 目前属于 UI 原型，不是正式入口。

这样做的目的很直接：把项目从“什么都塞一点”收回到“测量、判断、校正、导出”这条线。

## 5. 代码分层怎么理解

可以把当前项目理解成四层：

1. 采集层
   - STM32 固件 + 串口协议 + 串口读取线程。

2. 分析层
   - `filter_analysis.py`
   - `ai_diagnosis.py`
   - `control_compensation.py`

3. 展示层
   - `app.py`
   - `plotting.py`
   - `diagnosis_view.py`

4. 导出层
   - `report_export.py`

历史上加过的一些实验模块，可以理解成“旁路工具”，不影响主链路。

## 6. 最终建议

如果你拿这个项目去做演示，最好只讲三句话：

1. 它能把 STM32 的实测频响变成 Bode / Nyquist / 稳定性结论。
2. 它能自动解释电路类型、阶次和关键频率。
3. 它能按自动控制原理给出校正器建议。

其余功能都作为辅助工具看待，这样项目边界会清楚很多。
